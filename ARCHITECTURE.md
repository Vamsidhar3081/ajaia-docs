# Architecture Notes

## What I Prioritized and Why

### 1. Editor First
The editing experience is the product's core value. I chose **TipTap** (ProseMirror-based) because it gives a real, production-grade rich-text editing experience with a clean React API. It handles formatting, headings, lists, undo/redo, and is extensible. The alternative (contentEditable div + manual formatting) would've been brittle and taken longer to get right.

### 2. SQLite Over Postgres/Supabase
SQLite via `better-sqlite3` requires zero setup — no Docker, no connection strings, no external service. For a local-first demo, this is the right tradeoff. The schema is designed with foreign keys and ON DELETE CASCADE so migrating to Postgres later is straightforward (standard SQL).

### 3. JWT Auth (Stateless)
JWT tokens stored in localStorage. Simple, works with any frontend, no session store needed. Tokens expire in 7 days. For production: add refresh tokens and httpOnly cookies.

### 4. Autosave with Debounce
Documents autosave 1.5 seconds after the last keystroke (debounced). The save status indicator (✓ Saved / ⏳ Saving / ● Unsaved) gives users real-time feedback without being annoying. This mirrors Google Docs' behavior at 1/100th the complexity.

### 5. File Import Creates New Documents
Uploaded files become new editable documents rather than "attachments." This is more product-relevant — users import content they want to continue working on. The alternative (attaching files to existing docs) would require more UI complexity for less value.

## What I Cut (and Why)

| Feature | Reason Cut |
|---------|------------|
| Real-time collaboration (WebSockets) | Would require 3x the backend work; last-write-wins is fine for the demo |
| Email notifications on share | Requires SMTP/3rd party service; adds setup friction |
| Document version history | SQLite schema supports it (add a `document_versions` table), but the UI/UX takes time |
| Role-based permissions beyond view/edit | The current model (owner/edit/view) covers 90% of real use cases |
| Image uploads in documents | TipTap supports it; cut to stay in timebox |

## Stack

| Layer | Choice | Reason |
|-------|--------|--------|
| Frontend framework | React 18 | Ecosystem, TipTap support, familiar |
| Editor | TipTap 2 | Best React rich-text library, ProseMirror-based |
| Routing | React Router 6 | Standard |
| HTTP client | Axios | Interceptors, cleaner API than fetch |
| Backend | Express.js | Minimal, fast to write |
| Database | SQLite (better-sqlite3) | Zero setup, synchronous API is simpler |
| Auth | JWT + bcryptjs | Stateless, simple |
| File parsing | Mammoth (docx), built-in (txt/md) | Mammoth is the best Node docx converter |
| Styling | Custom CSS with design tokens | Full control, no Tailwind build step needed |

## Data Model

```sql
users (id, email, name, password_hash, created_at)
documents (id, title, content_html, owner_id, created_at, updated_at)
document_shares (id, document_id, shared_with_id, permission, created_at)
```

The `content` column stores HTML generated by TipTap. This is intentional — it preserves formatting exactly and is trivially renderable in the browser without re-parsing.

## Security Considerations (for production)

- Passwords hashed with bcrypt (cost 10)
- JWT secret should be rotated and stored in environment variables
- File uploads validated by extension + multer; in production, add MIME type sniffing
- SQLite is single-writer; for concurrent writes at scale, switch to Postgres
- Rate limiting should be added to auth endpoints
- CORS is currently open to the configured frontend URL only
